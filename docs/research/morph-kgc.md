## 1. What Morph-KGC Is

Morph-KGC (Arenas-Guerrero et al., *Semantic Web Journal* 15 (1), 2024, pp. 1–20) is a Python materializer for RDF knowledge graphs from [R2]RML mappings, built on top of pandas. It is the current practical SOTA for RDB-backed batch materialization. Version 2.10.0 (January 2026) is the latest release; the codebase is 99.8 % Python with 37 releases and 1 577 commits.

Its two performance levers are **mapping partitioning** (the core algorithmic contribution) and a handful of SQL-level optimizations (redundant self-join elimination, IS-NOT-NULL pushdown, vectorized pandas ops, hash joins for referencing joins). This document focuses on the partition mechanism.

---

## 2. The Invariant: Formal Foundation

The partition algorithm rests on one key concept — the **invariant** of a term map.

> The invariant *I* of a term map *T* is **the longest common initial part of all the RDF terms that can be generated by T**, regardless of the input data.
> — [SWJ paper §3](https://journals.sagepub.com/doi/10.3233/SW-223135)

Concretely:

| Term map type | Invariant |
|---|---|
| `rr:constant "http://ex.org/A"` | `"http://ex.org/A"` (the whole value) |
| `rr:template "http://ex.org/s/{id}"` | `"http://ex.org/s/"` (prefix before the first `{`) |
| `rr:column "name"` | `""` (empty — no fixed prefix) |
| Blank node | `""` (always gets partition "0") |

Two term maps T1 and T2 are **disjoint** iff the sets of RDF terms they can generate are disjoint regardless of input data. This holds when:
1. They have different *T* types (e.g., IRI vs literal), or
2. Their invariants have no prefix relationship — i.e., neither `I1.starts_with(I2)` nor `I2.starts_with(I1)`.

A **mapping partition** is a group of TriplesMaps (more precisely, of individual RML rules / triples-map rows) whose combined generated triples are guaranteed to be disjoint from every other partition's triples. Each partition can therefore be materialized independently — no inter-partition deduplication is ever needed.

## 3. Partition Computation Algorithm

The implementation lives in [`src/morph_kgc/mapping/mapping_partitioner.py`](https://github.com/morph-kgc/morph-kgc/blob/main/src/morph_kgc/mapping/mapping_partitioner.py). The `MappingPartitioner` class exposes one public method — `partition_mappings()` — which routes to one of three strategies controlled by the `mapping_partitioning` config key.

### 3.1 Step 0: Invariant extraction (`_get_term_invariants`)

Before partitioning, one pass over the normalized mapping rules populates four new columns on the rules DataFrame:

- `subject_invariant`, `predicate_invariant`, `object_invariant`, `graph_invariant`

Each is computed from `get_invariant_of_template(template)`, which strips the template down to the static prefix before the first unescaped `{reference}`. Constant maps use the full value. These columns are later dropped.

### 3.2 Strategy A — `PARTIAL_AGGREGATIONS` (default)

The algorithm makes four independent passes, one per RDF position. For each position:

1. Sort rules lexicographically by that position's invariant.
2. Scan in order. Start with `current_group = 0`, `current_invariant = ""`.
3. For each rule: if `rule.invariant.startswith(current_invariant)`, keep the rule in `current_group`. Otherwise, `current_group += 1` and set `current_invariant = rule.invariant`.
4. Assign `subject_partition`, `predicate_partition`, `object_partition`, `graph_partition`.

Special cases:
- **Blank nodes**: always `partition = 0`, never merged.
- **Literals**: first sub-grouped by datatype/language-tag (`literal_type` column), then by invariant within each datatype.
- **Constant-only predicates / graphs**: use exact-equality comparison rather than `startswith`, because a constant map's invariant *is* the whole value — no risk of prefix ambiguity.

Final partition label: `"{S}-{P}-{O}-{G}"` — a string key. Rules sharing this string belong to the same partition.

**Complexity**: O(n log n) per position for sorting, O(n) for the scan. Four passes → O(n log n) total.

### 3.3 Strategy B — `MAXIMAL` (optional, more groups, slower to compute)

Tries all 24 permutations of [S, P, O, G] position orderings. For each permutation, `_generate_maximal_partition_for_a_position_ordering()` applies the sequential partitioning logic from §3.2 but processes the four positions *jointly*, refining the partition assignment at each position. The permutation yielding the **maximum number of groups** is chosen.

To compute 24 orderings efficiently:

```python
pool = mp.Pool(self.config.get_number_of_processes())
mapping_partitions_dfs = pool.starmap(
    _generate_maximal_partition_for_a_position_ordering,
    zip([self.rml_df.copy()] * 24, position_orderings))
```

This is a brief parallelism burst at *partition-computation time* (before materialization), not at triple-generation time.

### 3.4 Strategy C — `NO_PARTITIONING`

Assigns every rule the label `"0-0-0-0"`. All rules land in one group; global deduplication (via a Python set) handles duplicates. Used as a performance baseline.

## 4. Parallelism Model: Partition-Level Process Pool

After partition labels are assigned, the materializer processes each group independently. The high-level flow in [`src/morph_kgc/materializer.py`](https://github.com/morph-kgc/morph-kgc/blob/main/src/morph_kgc/materializer.py):

```
for group_label, mapping_group_df in grouped_rules:
    _materialize_mapping_group_to_file(mapping_group_df, config, ...)
```

Parallelism is orchestrated by the *caller* of `_materialize_mapping_group_*`, dispatching groups across a `multiprocessing.Pool` (separate OS processes, bypassing the Python GIL). Each worker receives a copy of the mapping-group DataFrame and its own database connection.

**Configuration**:
- `number_of_processes = 1` → sequential mode; one group at a time; lowest memory footprint.
- `number_of_processes > 1` → parallel mode; default is `2 × cpu_count()`. Memory scales proportionally — each worker holds its partition's full dataset in memory simultaneously.

**Within each group**, the materializer iterates over rules (`for i, rml_rule in mapping_group_df.iterrows():`), generates triples, and accumulates into a Python `set` (`triples = set()`). The set provides automatic intra-partition deduplication at zero extra cost.

**Cross-partition deduplication is provably unnecessary**: because partitions are constructed from disjoint term maps, no triple can appear in two different partitions. This is the core claim that unlocks both the memory reduction and the embarrassingly-parallel dispatch.

## 5. SQL / RDBMS Data Access

Morph-KGC's database layer is in [`src/morph_kgc/data_source/relational_db.py`](https://github.com/morph-kgc/morph-kgc/blob/main/src/morph_kgc/data_source/relational_db.py):

- **Connectivity**: SQLAlchemy (optional since v2.7); supports MySQL, PostgreSQL, Oracle, MSSQL, MariaDB, SQLite, Snowflake, Databricks, Neo4j, KuzuDB.
- **Query generation**: table-based logical sources generate `SELECT col1, col2 FROM table WHERE col1 IS NOT NULL AND col2 IS NOT NULL`. SQL-query logical sources (`rr:sqlQuery`) pass through verbatim.
- **Data fetch**: `pd.read_sql_query(query, engine, coerce_float=False)` — pulls the entire result into a pandas DataFrame. No chunking or streaming.
- **Type inference**: post-fetch dtype assignment from `INFORMATION_SCHEMA` / `PRAGMA` / `ALL_TAB_COLUMNS` metadata maps SQL types → nullable pandas types (Int64, Float64, boolean, string).
- **Dialect adaptation**: backtick template → bracket (MSSQL), double-quote (PostgreSQL, SQLite, Oracle), no-quote (Databricks).
- **Referencing object maps** (R2RML join): handled in pandas via a hash join on the join-condition columns after fetching both parent and child tables. Multiple join conditions trigger a different pandas merge codepath, which is measurably slower (confirmed in [KROWN results](https://zenodo.org/records/10973892)).

**Redundant self-join elimination**: if two TriplesMaps reference the same logical source and the join condition references uniquely-identifying columns, the join is eliminated at mapping-normalization time — the child and parent fetch collapse to one query.

## 6. Benchmark Results

### 6.1 Hardware (SWJ paper experiments)

Intel Xeon Silver 4216 @ 2.10 GHz, 20 cores, 128 GB RAM, SSD SAS 12 GB/s. Timeout = 24 hours.

### 6.2 GTFS-Madrid-Bench — three engine configurations

| Config | Partitions? | Parallel? |
|---|---|---|
| Morph-KGC | No | No |
| Morph-KGC\_p | Yes | Sequential |
| Morph-KGC\_p+ | Yes | Parallel |

Scale factors tested: ×1, ×10, ×100, ×1000 (GTFS RDB data from Madrid subway network).

Key findings from [the paper's Figure 2 and Fig. 6](https://www.researchgate.net/figure/Total-execution-time-and-memory-consumption-peak-in-GTFS-Madrid-Bench-KGC-time-in_fig2_362982745):

- **Morph-KGC\_p+** clearly outperforms all other engines at every scale factor and every data format (CSV, JSON, SQL, XML).
- **RMLMapper** reaches timeout for data sizes larger than ×10 due to its global duplicate-elimination strategy — it maintains a global deduplication structure that grows without bound.
- **SDM-RDFizer** is the only *competitor* able to reach ×1000 due to its memory-compression operators; at small scales (×1, ×10) SDM-RDFizer achieves lower memory peaks than Morph-KGC\_p.
- **Morph-KGC\_p** (sequential partitions) reduces the memory peak significantly versus the no-partition baseline while keeping execution time roughly the same.
- Exact second/MB figures appear only in paper figures (not extractable from PDF in this session), but the ordering is clear and confirmed by multiple independent descriptions.

### 6.3 KROWN benchmark — failure modes

The [KROWN benchmark](https://github.com/kg-construct/KROWN) (kg-construct community) identifies three Morph-KGC weaknesses:

1. **Large cell sizes**: runs out of memory because entire tables are fetched into pandas in one shot (no streaming).
2. **Multiple join conditions**: execution time degrades because the multi-key pandas merge uses a different, slower code path than single-key merge.
3. **Many named-graph maps (≥10 per TriplesMap)**: partition labels collapse — the engine cannot apply its parallelism optimization and times out at 10–15 dynamic graph maps per mapping.

**OntopM** (Ontop in materialization mode) is unaffected by multiple join conditions because it pushes joins down to the SQL engine. RMLMapper times out with multiple conditions; SDM-RDFizer errors on multiple graph maps.

## 7. What semantic-fabric Should Adopt

### 7.1 Port the partition algorithm directly — it is substrate-agnostic

The invariant computation + prefix-scan algorithm operates on the *mapping IR* (a table of normalized rules), not on any data. In Rust:

- Represent each normalized TriplesMaps rule as a struct with four `Invariant` fields (subject, predicate, object, graph).
- `Invariant` is a `String` prefix (empty = no prefix; blank node flag handled separately).
- Implement `PARTIAL_AGGREGATIONS` as the default: four passes of sort-and-scan, O(n log n).
- Implement `MAXIMAL` as an optional exhaustive search: 24 permutations of [S, P, O, G], `rayon::par_iter()` over them, pick the permutation with the most partition groups.
- Assign partition label as a `(u32, u32, u32, u32)` tuple (cheaper than a string key).

This is pure logic with no I/O; a Rust port is straightforward.

### 7.2 Parallelism: rayon over partitions, DataFusion for data fetch

Replace the Python `multiprocessing.Pool` (separate OS processes, required to escape the GIL) with **rayon**:

```rust
partitions.par_iter().for_each(|partition| {
    let rows = fetch_rows_for_partition(&partition, &db_pool);
    let triples = generate_triples(&partition.rules, rows);
    writer.write_all(triples);   // partition-local, no locking needed
});
```

Because partitions are disjoint, writers can flush to separate output buffers and merge at the end — no lock contention.

For RDBMS data access, **DataFusion** (Apache Arrow, Rust-native) is the natural replacement for `pd.read_sql_query`:
- Use DataFusion's `TableProvider` trait to register the RDBMS table via JDBC/ODBC or native connectors.
- Push `IS NOT NULL` predicates and column projections down via `FilterPushdown` — Morph-KGC generates these manually; DataFusion makes them automatic.
- **Streaming execution**: DataFusion returns `RecordBatch` streams, eliminating Morph-KGC's KROWN weakness #1 (whole-table OOM). Each RecordBatch becomes a batch of triple-generation work dispatched through rayon.
- For referencing joins (parent/child tables), push the join to the SQL engine when possible (like OntopM); fall back to a hash join over Arrow batches, which is faster than pandas multi-key merge (KROWN weakness #2).

### 7.3 Duplicate elimination: trust the partitions, validate once

Within a partition, use a `BTreeSet<Triple>` or deduplicated Turtle serialization. Between partitions: no deduplication needed by the formal invariant argument — but implement a **debug/validation mode** (flag-guarded) that checks for cross-partition duplicates during test runs against the W3C R2RML test suite.

### 7.4 Naming-graph limitation — design opportunity

Morph-KGC fails on ≥10 dynamic graph maps because its partition labels can't distinguish group maps when graph invariants are template-based with many distinct values. In Rust, named-graph support is a first-class citizen in Oxigraph's quad model; partition labels should include the graph component from the start, and the invariant computation for graph term maps should be preserved.

### 7.5 Do not port: the pandas DataFrame representation of rules

Morph-KGC encodes normalized mapping rules as a pandas DataFrame (one row per triple pattern in each TriplesMap), then uses `groupby(mapping_partition)` to dispatch. In Rust, use a typed enum-based mapping IR — a `Vec<NormalizedRule>` grouped by partition into a `HashMap<PartitionLabel, Vec<NormalizedRule>>` — which is both more type-safe and avoids DataFrame overhead.

## 8. Relation to W3C R2RML / RML Test Suites and KROWN

Morph-KGC targets full R2RML conformance (W3C Recommendation) and scores high on the [W3C RDB2RDF test cases](https://www.w3.org/TR/rdb2rdf-test-cases/). RMLMapper holds the best conformance score for RML tests but fails at scale. semantic-fabric should target:

1. 100 % pass rate on the [W3C R2RML test suite](https://www.w3.org/2001/sw/rdb2rdf/test-cases/) (26 tests, all must pass — correctness gate).
2. GTFS-Madrid-Bench ×1000 RDB scenario — this is the published SOTA stress test where only SDM-RDFizer and Morph-KGC_p+ survive. Beating Morph-KGC_p+ on throughput is achievable via rayon + DataFusion streaming.
3. KROWN join-condition scenarios — a native SQL push-down approach should eliminate Morph-KGC's regression on multiple join conditions.

## 9. Open Questions

1. **Partition quality vs computation cost**: MAXIMAL partitioning tries 24 permutations but can take significant wall-clock time on large mapping documents. Is there a heuristic that achieves near-maximal groups in O(n log n)? The paper does not analyze this trade-off.
2. **Streaming compatibility**: if a partition's subject template produces IRIs that span multiple SQL result batches, can the invariant-based guarantee still hold for streaming triple emission? (Likely yes — invariants are data-independent by definition.)
3. **Join pushdown completeness**: the paper does not formalize which R2RML join conditions are safe to push to SQL. Ontop's [rewriting theory](https://journals.sagepub.com/doi/10.3233/SW-243580) is the reference for this.
4. **Named-graph partition handling at scale**: how to partition rules that use template-based named graphs without collapsing to a single group.
5. **KROWN update**: KROWN results are from a Zenodo 2024 snapshot; Morph-KGC 2.10.0 (Jan 2026) may have addressed some weaknesses — re-running is needed to establish the current baseline.
