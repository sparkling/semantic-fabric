It is not a W3C Standard and does not represent the consensus of the W3C or the RML Community Group; it is styled after the W3C R2RML Recommendation for familiarity only. It documents an extension that is implemented and test-gated in semantic-fabric. Where this specification reuses the RML-STAR vocabulary, it defines its own compiled semantics for those terms — a deliberate, documented divergence described in §9.

§Abstract

R2RML maps relational rows to RDF triples for virtual, non-materializing access (OBDA). RDF-star (RDF 1.2) adds quoted triples — triples that are themselves the subject or object of other triples — enabling statements about statements (provenance, certainty, temporal validity). This specification defines R2RML-star: a minimal extension letting a mapping author declare that a triples map's subject or object represents a quoted triple, and defining how a SPARQL-star query <<?s ?p ?o>> is answered against relational data. The extension compiles quoted triples to the W3C RDF 1.2 Interoperability basic encoding — plain RDF reachable by an ordinary SQL rewrite — rather than to a native triple term, because a virtualizing SQL backend has no atomic triple-term value to store or scan.

1.Introduction

A quoted triple <<s p o>> lets an RDF graph carry statements about other statements. The canonical example is provenance: not merely "Alice is 34", but "the claim that Alice is 34 was asserted by the 2026 census." In a native RDF-star store, the quoted triple is a first-class term. In an OBDA setting the data lives in relational tables and is never materialized, so there is no place to store a triple term and no atomic value for a SQL SELECT to scan.

R2RML-star resolves this by compiling a quoted triple, at both mapping and query time, into the RDF 1.2 Interoperability basic encoding: a synthetic identifier plus four plain triples that name its subject, predicate and object. Plain triples are exactly what R2RML already maps and what a SQL rewrite already reconstructs, so the extension is additive — it introduces no new execution machinery.

Note — why not native triple terms RDF 1.2 Interoperability defines the basic encoding precisely as the plain-RDF form a system without native triple-term storage can round-trip. A survey of seven OBDA/virtualization engines and the 2023–2026 literature (ADR-0028 §G) found no prior art for RDF-star over live SQL rewriting; the basic encoding is the only shape a query-time SQL rewrite can reconstruct.

2.Terminology and Conformance

The key words MUST, MUST NOT, SHOULD and MAY are to be interpreted as in RFC 2119.

TermMeaning
quoted tripleAn RDF-star triple appearing as the subject or object of another triple, written <<s p o>>.
star mapA term map (§4) that generates the identity of a quoted triple instead of an ordinary term.
quoted triples mapThe triples map whose (subject, predicate, object) is the quoted triple.
proposition identifierThe synthetic IRI that stands in for a quoted triple in the basic encoding (§5.1).
basic encodingThe plain-RDF reification of a quoted triple: rdf:PropositionForm + rdf:propositionForm{Subject,Predicate,Object} (§5.2).

A conforming R2RML-star processor MUST compile every rml:starMap it accepts to the basic encoding of §5, MUST reject the constructs of §8 at mapping-load time with an explicit error (never a silent wrong result), and MUST answer a SPARQL-star quoted-triple pattern by the rewrite of §7.

3.The R2RML-star Vocabulary

R2RML-star reuses the RML-STAR vocabulary verbatim at the authoring surface (namespace http://semweb.mmlab.be/ns/rml#, prefix rml:):

IRIKindRole
rml:starMappropertyAttached to a subject or object map; its value is a star-map node declaring a quoted triple.
rml:quotedTriplesMappropertyThe quoted triples map, asserted (its plain triple is also produced — §6).
rml:nonAssertedTriplesMappropertyThe quoted triples map, non-asserted (its plain triple is suppressed — §6).

Compilation targets the RDF 1.2 basic-encoding vocabulary (namespace http://www.w3.org/1999/02/22-rdf-syntax-ns#, prefix rdf:):

IRIRole
rdf:PropositionFormThe class of a proposition identifier.
rdf:propositionFormSubjectLinks the identifier to the quoted triple's subject.
rdf:propositionFormPredicateLinks the identifier to the quoted triple's predicate.
rdf:propositionFormObjectLinks the identifier to the quoted triple's object.
rdf:reifiesThe RDF 1.2 native reification predicate that SPARQL-star bare syntax desugars to at query time (§7.1).

4.Star Maps

A star map is an rml:starMap node nested inside a subject map or an object map. Its rml:quotedTriplesMap (or rml:nonAssertedTriplesMap) references the triples map whose subject/predicate/object is the quoted triple.

4.1Subject-position star maps

When the star map sits in the subject map, the enclosing triples map's subject is the quoted triple. The four basic-encoding triples are emitted onto that same triples map (they share its synthetic subject).

<#PersonAgeAssertion>
    a rr:TriplesMap ;
    rr:logicalTable [ rr:tableName "census_row" ] ;
    rr:subjectMap [
        rml:starMap [ rml:quotedTriplesMap <#PersonAge> ]
    ] ;
    rr:predicateObjectMap [
        rr:predicate ex:assertedBy ;
        rr:objectMap [ rr:constant ex:CensusRecord2026 ]
    ] .

4.2Object-position star maps

When the star map sits in an object map, the quoted triple is the object of the enclosing triple. Because the proposition identifier is the object here (not the enclosing subject), the four basic-encoding triples are emitted as a separate synthetic triples map keyed on that identifier.

Note Both positions are supported. The mapping is identical in spirit; only where the four basic-encoding triples attach differs. A quoted triple MUST NOT appear in predicate position — see §8.

5.Compilation to the Basic Encoding

5.1The synthetic proposition identifier

R2 The proposition identifier is compiler-derived, never author-supplied. It is a deterministic IRI template of the form urn:sf-star:<predicate-slug>:{cols…}, built from the quoted triple's predicate and the column references of its subject and object term maps. Determinism is structural: the same underlying row always expands to the same identifier, so the one reification per quoted triple invariant holds by construction rather than by author discipline. A collision-safe delimiter is used, and no run-time hash function is required (keeping the identifier backend-portable).

5.2The four basic-encoding triples

R1 For a quoted triple (s, p, o) with proposition identifier I, the processor emits:

I  rdf:type                     rdf:PropositionForm .
I  rdf:propositionFormSubject    s .
I  rdf:propositionFormPredicate  p .
I  rdf:propositionFormObject     o .

The subject s, predicate p and object o are copied verbatim from the quoted triples map's own term maps, so they generate exactly the terms they would in an ordinary mapping. No new SQL construction is introduced: these are ordinary predicate-object maps.

6.Asserted and Non-Asserted Triples Maps

Whether the quoted triple is also a plain, independently-matchable triple is chosen by which property references the quoted triples map:

DeclarationPlain triple s p o emitted?Basic encoding emitted?
rml:quotedTriplesMap (asserted)YesYes
rml:nonAssertedTriplesMap (non-asserted)NoYes

A quoted triples map referenced only by non-asserted star maps is suppressed from the plain output; an assertion by any star map anywhere reinstates it.

7.Query Semantics (SPARQL-star)

7.1The rdf:reifies desugaring

A SPARQL-star parser conformant to RDF 1.2 desugars the everyday bare syntax <<s p o>> into a fresh blank node plus a triple _:b rdf:reifies <<( s p o )>> — the RDF 1.2 native reification form. Since no relational mapping ever asserts rdf:reifies, a rewrite that merely translated the quoted triple in place would leave an unmatchable rdf:reifies triple in the pattern and every bare-syntax query would silently return zero rows. A conforming processor MUST therefore recognize and elide the rdf:reifies wrapper.

7.2The pattern rewrite

R1 Quoted-triple patterns are rewritten to the basic encoding at the algebra level, before query planning, so no downstream stage encounters a quoted triple. For each quoted triple <<s p o>> the processor introduces a fresh identity variable ?id and the four patterns:

?id  rdf:type                     rdf:PropositionForm .
?id  rdf:propositionFormSubject    s .
?id  rdf:propositionFormPredicate  p .
?id  rdf:propositionFormObject     o .

R2 The wrapper X rdf:reifies <<( s p o )>> is elided: the rdf:reifies triple is dropped and the four patterns are bound on X directly (where X may be a blank node the parser introduced, or an author variable). R5 An identity position always binds the synthetic proposition IRI — v1 never fabricates a native triple term in results.

8.Scope and Restrictions (v1)

The following are rejected as explicit errors — never silent empties, never silent wrong values:

RestrictionRationale
R3No nesting. A quoted triple whose own subject or object is itself a quoted triple is rejected.Single-level only; nesting needs SQL-view precomputation.
R4No predicate position. A quoted triple in predicate position is rejected.Consistent with RML-STAR and with the SPARQL grammar (predicate is a name, not a term).
—Cross-source (mapping): a quoted triples map whose logical source differs from the outer map is rejected in v1.The join case is deferred; single-row derivation only.
—Query boundaries: a ground quoted triple in VALUES, the TRIPLE()/SUBJECT()/PREDICATE()/OBJECT()/isTRIPLE() functions, a CONSTRUCT template quoting a triple, and a quoted pattern at a property-path endpoint each return an explicit 501-class error in v1.Each needs machinery outside the pattern rewrite; declared, tested boundaries rather than silent gaps.

9.Divergence from RML-STAR

Important — documented semantic divergence R2RML-star reuses RML-STAR's vocabulary but defines its own compiled semantics: an rml:starMap compiles to the plain-RDF basic encoding, never to a native RDF-star/RDF 1.2 triple term. An author or portability tool that assumes rml:starMap asserts a native triple term portable to any RML-STAR processor will be wrong about what querying the result returns. This is a deliberate choice for a virtualizing engine (which has no triple-term storage), and it is stated wherever this vocabulary is used.

10.Worked Example

A relational table of census claims, mapped so each age claim is quoted and attributed to a record.

Relational source — census_row 
person_id | age
----------+----
     1042 |  34
     2087 |  29
R2RML-star mapping 
@prefix rr:  <http://www.w3.org/ns/r2rml#> .
@prefix rml: <http://semweb.mmlab.be/ns/rml#> .
@prefix ex:  <http://example.com/> .

<#PersonAge>
    a rr:TriplesMap ;
    rr:logicalTable [ rr:tableName "census_row" ] ;
    rr:subjectMap [ rr:template "http://ex.org/person/{person_id}" ] ;
    rr:predicateObjectMap [
        rr:predicate ex:hasAge ;
        rr:objectMap [ rr:column "age" ]
    ] .

<#PersonAgeAssertion>
    a rr:TriplesMap ;
    rr:logicalTable [ rr:tableName "census_row" ] ;
    rr:subjectMap [ rml:starMap [ rml:quotedTriplesMap <#PersonAge> ] ] ;
    rr:predicateObjectMap [
        rr:predicate ex:assertedBy ;
        rr:objectMap [ rr:constant ex:CensusRecord2026 ]
    ] .
SPARQL-star query 
PREFIX ex: <http://example.com/>
SELECT ?person ?age ?src WHERE {
    << ?person ex:hasAge ?age >> ex:assertedBy ?src .
}
Result 
?person                     | ?age | ?src
----------------------------+------+---------------------------------
http://ex.org/person/1042   | 34   | http://example.com/CensusRecord2026
http://ex.org/person/2087   | 29   | http://example.com/CensusRecord2026

The engine answered the quoted pattern by joining the basic-encoding rows the mapping compiles to — no triple term was materialized, no data left the database.
