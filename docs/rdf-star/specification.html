§Abstract

R2RML maps relational rows to RDF triples for virtual, non-materializing access (OBDA). RDF-star (RDF 1.2) adds quoted triples — triples that are themselves the subject or object of other triples — enabling statements about statements (provenance, certainty, temporal validity), expressed either as reified triples (<<s p o>>, asking about the reifier of a statement) or as bare triple terms (<<( s p o )>>, the statement itself as a value). This specification defines R2RML-star: an extension letting a mapping author declare that a triples map's subject or object represents a quoted triple, and defining how SPARQL-star patterns — of either kind — are answered against relational data. The virtualized graph is a native RDF 1.2 graph: reifiers, rdf:reifies triples and triple terms are real, queryable, projectable RDF 1.2 values at every surface a client sees — SELECT bindings, CONSTRUCT output, serializations. Underneath, the extension compiles quoted triples to the W3C RDF 1.2 Interoperability basic encoding — plain RDF reachable by an ordinary SQL rewrite — used purely as an internal wire format below the SQL line, because a virtualizing SQL backend has no atomic triple-term value to store or scan. Soundness and completeness are defined, and mechanically verified, as answer equivalence with a native SPARQL 1.2 evaluator run over the decoded graph (ADR-0032).

1.Introduction

A quoted triple <<s p o>> lets an RDF graph carry statements about other statements. The canonical example is provenance: not merely "Alice is 34", but "the claim that Alice is 34 was asserted by the 2026 census." In a native RDF-star store, the quoted triple is a first-class term. In an OBDA setting the data lives in relational tables and is never materialized, so there is no place to store a triple term and no atomic value for a SQL SELECT to scan.

R2RML-star resolves this by compiling a quoted triple, at both mapping and query time, into the RDF 1.2 Interoperability basic encoding: a synthetic identifier plus four plain triples that name its subject, predicate and object. Plain triples are exactly what R2RML already maps and what a SQL rewrite already reconstructs, so the extension is additive — it introduces no new execution machinery.

Note — encoding is an implementation detail, not what you query A SQL row has no atomic triple-term value to store or scan, so underneath the engine still represents each quoted triple as a synthetic identifier plus plain triples — the RDF 1.2 Interoperability basic encoding, reachable by an ordinary SQL rewrite. But the engine decodes at every query boundary (§7.6): a client never has to reason about the encoding to get a correct answer, only to know it exists below the SQL line. A survey of seven OBDA/virtualization engines and the 2023–2026 literature (ADR-0028 §G) found no prior art for RDF-star over live SQL rewriting either way.

2.Terminology and Conformance

The key words MUST, MUST NOT, SHOULD and MAY are to be interpreted as in RFC 2119.

TermMeaning
quoted tripleAn RDF-star triple appearing as the subject or object of another triple, written <<s p o>> (reified) or <<( s p o )>> (a triple term) — see §7.1.
star mapA term map (§4) that generates the identity of a quoted triple instead of an ordinary term.
quoted triples mapThe triples map whose (subject, predicate, object) is the quoted triple.
proposition identifierThe synthetic urn:sf-star:pf:… IRI that stands in, in the basic encoding, for a quoted triple's triple term — a pure function of its (s,p,o) shape (§5.1).
reifier identifierThe synthetic urn:sf-star:r:… IRI that stands in, in the basic encoding, for the reifier of a subject-position quoted triple — one per star-map declaration (§5.1).
basic encodingThe plain-RDF reification of a quoted triple: rdf:PropositionForm + rdf:propositionForm{Subject,Predicate,Object}, plus rdf:reifies where a reifier exists (§5).

A conforming R2RML-star processor MUST compile every rml:starMap it accepts to the basic encoding of §5, MUST reject the constructs of §8 at mapping-load time with an explicit error (never a silent wrong result), and MUST answer a SPARQL-star quoted-triple pattern by the rewrite of §7.

4.Star Maps

A star map is an rml:starMap node nested inside a subject map or an object map. Its rml:quotedTriplesMap (or rml:nonAssertedTriplesMap) references the triples map whose subject/predicate/object is the quoted triple. Authoring syntax is identical regardless of position; what differs is what the star map compiles to (§5).

4.1Subject-position star maps

When the star map sits in the subject map, the enclosing triples map's subject is native RDF 1.2's reifier of the quoted triple, not the triple term itself (Concepts §1.5: a reifier and the proposition it reifies are distinct). The processor emits, per row:

  • the outer triples map's own predicate-object maps, now hanging off the reifier identifier as their subject — this is where author annotations (like ex:assertedBy below) land, exactly the native RDF 1.2 annotation shape;
  • one injected rid rdf:reifies pfid predicate-object map, linking the reifier to the proposition identifier (§5.3);
  • the four basic-encoding triples describing the proposition identifier (§5.2), on a standalone triples map shared by every reference to the same quoted shape (§5.1).
<#PersonAgeAssertion>
    a rr:TriplesMap ;
    rr:logicalTable [ rr:tableName "census_row" ] ;
    rr:subjectMap [
        rml:starMap [ rml:quotedTriplesMap <#PersonAge> ]
    ] ;
    rr:predicateObjectMap [
        rr:predicate ex:assertedBy ;
        rr:objectMap [ rr:constant ex:CensusRecord2026 ]
    ] .

This is the same mapping v1 authors already wrote; only its compiled shape changed — §10 walks both identifiers for this exact example.

4.2Object-position star maps

When the star map sits in an object map, the quoted triple is the object of the enclosing triple — legal per RDF 1.2 Concepts §3.1, and the only position a triple term may occupy. The object is the proposition identifier directly; a reifier stands for an assertion about a triple, and here the quoted triple is just a value the outer map's own predicate-object map happens to point at — not a statement the outer map is making about it — so there is nothing to reify (RDF 1.2 Concepts §3.1): no rdf:reifies triple or reifier identifier is minted. The four basic-encoding triples are emitted onto the same standalone description map §4.1 uses for the identical shape — deduplicated, whichever position reaches it first.

Note Both positions are supported, including arbitrarily deep nesting on the object side — a quoted triple whose own object is itself a quoted triple (§5.1). Subject-side nesting and predicate-position quoting both stay load-time errors — see §8, citing RDF 1.2 Concepts §3.1.

4.3Overriding the reifier: rml:reifierMap

R2 The default reifier identifier is deterministic and compiler-derived, keyed on the declaring star map's own triples-map id (§5.1). A subject-position star map MAY instead supply rml:reifierMap — an ordinary term map, constrained to IRI or blank-node terms (RDF 1.2 requires a reifier to be a non-literal) — to control the reifier's own identity: 

rr:subjectMap [
    rml:starMap [
        rml:quotedTriplesMap <#PersonAge> ;
        rml:reifierMap [ rr:template "http://ex.org/reifier/{person_id}" ]
    ]
] .

rml:reifierMap is rejected at load time on an object-position star map: object position mints no reifier to override (§4.2).

5.Compilation to the Basic Encoding

5.1The two identifier families

R2 Two distinct, deterministic, compiler-derived, never author-supplied (barring §4.3) IRI families realize RDF 1.2's own split between a proposition and its reifiers (Concepts §1.5, Semantics §5):

FamilyTemplate shapeKeyed onStands in for
Proposition idurn:sf-star:pf:<predicate-slug>|<subject-component>|<object-component>|The quoted (s,p,o) shape, over each component's full rendered formThe triple term itself
Reifier idurn:sf-star:r:<declaring-map-slug>|{cols…}|The declaring star map, plus the same columns as the proposition id it reifiesOne reifier of that proposition

A proposition id is a pure function of the quoted shape: every reference to the same (s,p,o) shape — from anywhere in the mapping, subject position or object position — expands to the identical identifier for the identical row. Each of the subject and object components is the term's full rendered lexical form, not merely its referenced columns:

  • a compile-time term-kind tag — I (IRI), B (blank node), L (plain literal), L^^<datatype>, or L@<language> — folded in as fixed text ahead of the component, since a term map's kind, datatype and language never vary per row;
  • an rr:template term's entire segment list — literal prefixes and suffixes as well as columns — spliced in verbatim, exactly as template expansion renders the real term elsewhere in the same mapping, so two templates over the same column(s) with different fixed text can never collide; an rr:column term contributes only its own per-row value, since there is no prefix or suffix to lose;
  • an rr:constant term's own canonical form, percent-encoded directly into the id — it is never expanded anywhere else, so nothing else fixes its representation;
  • every per-row column value, percent-encoded exactly as template expansion always percent-encodes an IRI segment (R2RML §7.3's iunreserved set) — | itself falls outside that set, so a | surviving into the finished id is always one of the id's own delimiters, never smuggled-in column content.

This makes the proposition id an injective function of the quoted (s,p,o) shape, realizing Semantics §5's co-identity mapping IT(s,p,o) structurally: id-equality now implies decoded-triple-equality, so two differently-templated star maps that share a predicate and happen to produce the same raw column values can no longer mint the same id for genuinely different triples. A reifier id is instead keyed on which star map declared it: two different subject-position star maps quoting the same shape mint two distinct reifiers of the one (deduplicated) proposition — Concepts §1.5's "there can be multiple, distinct reifiers related to the same abstract proposition." Both families use structural determinism, not a run-time hash: the same row always expands to the same string, so identifiers are backend-portable and reproducible across independent runs.

Nesting. When a quoted triple's own object is itself a quoted triple (§4.2), the inner quote's own proposition id is itself an rr:template term to the outer one — so the same splice-in-verbatim rule above embeds the inner shape's entire identity into the outer id, rather than flattening to bare column names, at any nesting depth.

5.2The four basic-encoding triples

R1 For a quoted triple (s, p, o) with proposition identifier pfid, the processor emits, onto a standalone triples map keyed on pfid (§4):

pfid  rdf:type                     rdf:PropositionForm .
pfid  rdf:propositionFormSubject    s .
pfid  rdf:propositionFormPredicate  p .
pfid  rdf:propositionFormObject     o .

The subject s, predicate p and object o are copied verbatim from the quoted triples map's own term maps, so they generate exactly the terms they would in an ordinary mapping — except where o is itself a nested quote (§4.2), in which case it is that inner quote's own pfid. These triples are always keyed on the proposition id, never the reifier id: a reifier reifies the proposition, it does not have its own description. No new SQL construction is introduced beyond ordinary predicate-object maps. When the declaring star map sits under an rr:graphMap — subject or object position alike — this standalone triples map's own subject map inherits the same effective graphs as the quoting map, so the four triples land in the identical named graph as the quote itself, not the default graph (§7.8).

5.3The rdf:reifies triple

A subject-position star map (§4.1) additionally emits one predicate-object map onto the outer triples map: 

rid  rdf:reifies  pfid .

where rid is the reifier identifier (§5.1, or an rml:reifierMap override, §4.3) — now the outer triples map's own subject — and pfid is the proposition identifier of the quoted shape. This triple is the native RDF 1.2 reification link (Concepts §1.5) and is what §7's query rewrite matches against; it is never emitted for an object-position star map, which mints no reifier (§4.2).

6.Asserted and Non-Asserted Triples Maps

Whether the quoted triple is also a plain, independently-matchable triple is chosen by which property references the quoted triples map:

DeclarationPlain triple s p o emitted?Basic encoding emitted?
rml:quotedTriplesMap (asserted)YesYes
rml:nonAssertedTriplesMap (non-asserted)NoYes

A quoted triples map referenced only by non-asserted star maps is suppressed from the plain output; an assertion by any star map anywhere reinstates it.

7.Query Semantics (SPARQL-star)

7.1The matching law

R3 SPARQL-star has two distinct node kinds wherever a quoted triple appears, and they are never interchangeable:

SyntaxNode kindAsks about
<< s p o >> (bare)ReifiedTripleReifiers — desugars to _:b rdf:reifies <<( s p o )>>, §7.2
<< s p o ~ r >> (explicit reifier)ReifiedTripleReifiers — same as bare, with r in place of the fresh blank node
s p o {| … |} (annotation)ReifiedTriple, plus the plain tripleReifiers — and asserts s p o itself (the only sugar that does both)
<<( s p o )>> (parenthesized)TripleTermTriple terms — the statement as a value, in place

Bare and explicit-reifier sugar do not assert s p o and do not match an unreified object-position triple term — they match only a genuine rdf:reifies statement (§5.3, §7.2). Annotation sugar is the one spelling that both asserts and reifies in one step, mirroring Turtle 1.2's own data-side table exactly. A query that writes << ?s ?p ?o >> ?p2 ?o2 is asking "who/what reifies this statement, and what do they say about it?"; one that writes ?x ?p2 <<( ?s ?p ?o )>> is asking "is this exact statement, as a value, the object of some other triple?" — different questions, different answers.

7.2The pattern rewrite

Quoted-triple patterns are rewritten to the basic encoding at the algebra level, before query planning, so no downstream stage encounters a quoted triple.

Reified-triple patterns — X rdf:reifies TT, whatever sugar produced them — are rewritten with no elision: the rdf:reifies triple is kept, as X rdf:reifies ?pf (a fresh variable), joined against the four basic-encoding patterns on ?pf: 

X    rdf:reifies                   ?pf .
?pf  rdf:type                      rdf:PropositionForm .
?pf  rdf:propositionFormSubject    s .
?pf  rdf:propositionFormPredicate  p .
?pf  rdf:propositionFormObject     o .

This matches only rows where a real rdf:reifies triple exists (§5.3) — i.e. a subject-position star map's output. Projecting X binds the reifier identifier (§5.1); the four-pattern join is what actually locates it.

Triple-term patterns in object position — ?x pred <<( s p o )>> — rewrite to ?x pred ?pf plus the same four basic-encoding patterns on ?pf, matching any triple-term object regardless of whether it is also reified.

Nesting (§4.2, §5.1): where o is itself a quoted-triple pattern, the rewrite recurses bottom-up — the innermost quote mints its own fresh identity variable and four patterns first, spliced in as the enclosing level's propositionFormObject — to arbitrary depth. Subject-side nesting never reaches this rule; see §7.3.

7.3Subject-position patterns are statically empty

R4 SPARQL 1.2 §18.1.3 is explicit: "a triple pattern that has another triple pattern in its subject position will fail to match on any RDF graph." A triple-term pattern — bare or parenthesized — in subject position is legal to write and compiles to a statically empty group: never an error, never a match, checked before the predicate is even inspected (so it also covers a nested quote whose own subject is a triple term, reached through an outer quote's object chain).

7.4Nesting, VALUES, and the five functions

R5 A ground triple term in VALUES decomposes into its three components, joined against the basic-encoding patterns wherever the variable is also matched elsewhere; where the variable is only projected, the ground term is reprojected directly — including nested ground terms, decomposed recursively. Rows mixing a triple-term cell and a plain-value cell for the same VALUES column resolve at the top level of a SELECT (the block row-partitions into a union, each row realizing per its own shape — see §8.2); wherever the mixed variable is observed by a function, equality, or template, the explicit 501 remains.

The five SPARQL 1.2 triple-term functions (§17.4.6) are implemented by engine-totality: relational data can never itself contain a native triple term, so every triple-term-valued expression in a supported query is one the rewrite already knows about statically. TRIPLE(e1,e2,e3) builds a composed value (legality errors enforced per §17.4.6); SUBJECT/PREDICATE/OBJECT on a known composed value return its component, and on anything else raise the spec's ERROR — row-eliminating in FILTER, genuinely unbound (not a wrong value) in BIND; isTRIPLE never errors, returning a constant true/false.

7.5Equality and ORDER BY

sameTerm over two composed values is component-wise syntactic conjunction. = is recursive component sameValue equality (§17.4.2.2), propagating a component error; comparing a composed value against a non-composed one is statically false, never an error. ORDER BY places triple terms in SPARQL's highest sort category (§15.1); the spec leaves ordering among triple terms undefined, so the engine's deterministic serialization order is a permissible choice — consistent across repeated runs of the same query, not standardized across implementations.

7.6Native results

R1 Every visible surface speaks native RDF 1.2 reification form. A projected reified-triple-pattern variable binds the reifier's plain IRI (§5.1); a projected triple-term variable — from a triple-term pattern, a decomposed VALUES row, or a TRIPLE() call — binds a real triple term, decoded at the query boundary from the composed components the rewrite tracked all along. In SPARQL 1.2 Results JSON this is the standard recursive "type":"triple" binding shape (§10 walks a full example); Turtle 1.2 CONSTRUCT output serializes the same value as <<( … )>> in place. A CONSTRUCT template that instantiates a triple term in object position produces a genuine one; an instantiation landing it in a position RDF 1.2 forbids is silently dropped from the output graph, per SPARQL §16.2 — not an error.

7.7Set semantics: BGP-level deduplication

R6 R2RML's own output model is a graph — a set of triples — and SPARQL §18.3 evaluates a basic graph pattern against that set: a duplicate source row, or two candidate maps emitting the identical triple, must not inflate the answer. This is engine-general, not star-specific (ADR-0034) — the mechanism dedups at the BGP-block boundary, below GROUP BY, before projection or UNION can introduce their own legitimate duplicates. It is documented here because it is exactly what already makes the four basic-encoding triples (§5.2) shared, not duplicated, across every reference to the same quoted shape, and because star's extra shared-variable join positions are where it is most easily triggered.

Two mechanisms, elided to zero SQL cost in the common case:

  • Within-branch (duplicate source rows). A branch whose joined tables are not all covered by a declared primary/unique key gets a SELECT DISTINCT wrap over its full read set. A well-keyed mapping — subjects templated on a primary key, the overwhelmingly common shape — needs no wrap at all: the key-coverage proof elides it, so the emitted SQL is byte-identical to before.
  • Cross-map (two maps, one triple). Where two candidate maps' arms are not provably disjoint, they pool through UNION (set) instead of UNION ALL (bag). Provably disjoint arms — different fixed template prefixes (the normal case for two genuinely different shapes), a mismatched term kind or language tag, or a mismatched literal datatype (2026-07-20) — already cannot emit the same triple, so UNION ALL stays set-correct and no pooling is introduced.

Where an arm's own template is not injective — a Literal or BlankNode template that can render the same value from different rows — SQL DISTINCT cannot be trusted to equal term-level dedup, so an unkeyed, non-joined (standalone) relation instead dedups in Rust, on the fully reconstructed solution tuple. This does not extend to non-injective IRI templates, which stay a 501 refusal: unlike a literal or blank node, an IRI template can always be made injective by adding a separator between its column references, so the refusal is an addressable mapping-design issue, not an engine limitation. One further, PostgreSQL-only 501 remains: pooling a float-typed component against a differently-typed one in the same UNION risks silent lexical drift in PostgreSQL's own float-to-text formatting, so it is refused rather than risked.

CONSTRUCT output is a set under the same reading (§16.2): instantiating the same triple from two different result rows emits it once, not twice — applied at the SQL level whenever the template's variables are a strict subset of the pattern's own and the template is blank-node-free. This extends across branches too: two different candidate maps instantiating the identical triple from different solutions still emit it once, via a dedup set carried across solutions at execution time, not only the single-branch SQL-level case above.

7.8Named graphs: GRAPH ?g

R7 A constant-graph pattern, GRAPH <g> { P }, already answers under ordinary R2RML: a triple lands in named graph g iff its predicate-object map's effective graph (its own rr:graphMaps, else the subject map's; none ⇒ default graph, R2RML §4.6) produces g. The variable form, GRAPH ?g { P }, is answered by per-branch graph enumeration: each candidate predicate-object map contributes one translation branch per effective graph map it declares — a constant graph map becomes a bound term for ?g, a template or column graph map becomes an ordinary derived binding — and a map with no graph map at all is excluded from the branch set entirely, since SPARQL §13.3 ranges GRAPH ?g over named graphs only, never the default graph. No quad store and no dataset materialization are needed: ?g is just an ordinary variable with per-branch definitions, so two patterns sharing one ?g inside a GRAPH ?g block join on the same graph for free, through the same unification machinery that already joins any shared variable — projection of ?g, FILTER(?g = …), and VALUES ?g come along the same way.

Star annotations inside named graphs. The standalone description triples map carrying the four basic-encoding triples (§5.2) is compiled once and shared by every reference to a given quoted shape (§4.1). Its own predicate-object maps inherit the quoting map's effective graphs (§5.2): a star map declared under an rr:graphMap — subject or object position, symmetrically — produces a description map whose triples live in that same named graph, so GRAPH ?g { << s p o >> ex:p ?v } binds ?g and answers exactly as an equivalent non-star pattern would.

Property paths under GRAPH ?g compile as a union over the mapping's declared constant named graphs — one GRAPH <gi> arm per declared constant graph (each already answerable on its own, §8.1), with ?g = gi bound per arm. A path under a graph map that is a template or column is a pinned 501: the graph set is row-dependent, not enumerable at translate time — see §8.2.

8.Scope and Restrictions

Nearly every boundary v1 declared a 501-class error is now implemented and test-gated (ADR-0032). What follows splits what changed from what genuinely remains — every remaining item is still an explicit error, or an explicit empty match, never a silent wrong answer.

The wrong-graph fix shares one two-pass predicate-resolution rewrite across both engines.
—Uniform composed-ness — the FILTER carve-out (2026-07-19). A FILTER over a mixed-composedness union or VALUES partition, sitting at the SELECT top level or inside an EXISTS/NOT EXISTS body, now resolves per arm — each arm's own static shape decides its own rows — instead of collapsing to one answer for the whole union. Bare projection of a mixed variable already resolved per-arm (unchanged). Still refused: a CONSTRUCT template consuming the mixed variable, and a mixed union/VALUES nested under a further join.The FILTER case mirrors the rewrite's existing Filter-over-Union distribution; EXISTS/NOT EXISTS route through the same top-level rewrite. A CONSTRUCT template has no per-row arm to realize into and stays refused; nesting under a further join is outside the boundary the fix reaches.
—Whole-composed-variable equality — shape-mismatch residual mostly closed (2026-07-19). ?t1 = ?t2 already resolved same-structure templates and constants (pairwise column equality). Genuinely differently-shaped templates now resolve too, for IRI and plain/xsd:string components, by comparing each side's fully-rendered form as a dialect-correct SQL CONCAT — on PostgreSQL, SQLite and MySQL. The residual 501 narrows to typed-literal, language-tagged, and blank-node component classes, and to any other SQL dialect.A second, purely boolean-condition SQL rendering (SqlCond::TemplateEq) alongside the alignment-based path above — never SELECTed or reconstructed from, so term-construction lifting is untouched (ADR-0007). Restricted to term classes where lexical equality is term equality, and to dialects whose NULL-propagation through string concatenation matches R2RML §11 term-absence.
—Same-shape, cross-kind template equality — closed (2026-07-20). The same-structure fast path above (pairwise column equality) used to compare two templates of the same shape without checking term kind, so FILTER(?a = ?b) between a same-shape IRI template and a Literal template — or two Literal templates with different language tag or datatype — could wrongly answer non-empty. Same-shape, cross-kind or cross-datatype pairs now resolve to the provably-empty match they always were, for both the FILTER/= path and the JOIN path that already guarded this case upstream.The check now lives inside the shared alignment routine, so every caller gets it once: same kind and same normalized datatype/language ⇒ column equality; anything else ⇒ statically empty — sound and complete, since cross-kind or cross-datatype terms are never sameTerm.
—Path under a variable named graph, mapping-wide non-constant guard (ADR-0035, 2026-07-20). A property-path pattern inside GRAPH ?g { … } compiles as a union over the mapping's declared constant named graphs (§7.8) — sound and complete whenever every graph map in the mapping is constant. The moment the mapping declares even one template- or column-valued graph map anywhere — not necessarily on the path's own predicate — the path refuses.The graph set of a template/column graph map is row-dependent, not enumerable at translate time, so a per-predicate union isn't available for it; the guard is conservative and mapping-wide rather than narrowed to the predicates the path actually touches. Narrowing this to per-predicate relevance is a recorded, not-yet-implemented refinement.

9.Divergence from RML-STAR

The headline divergence is dissolved Through ADR-0032, every visible surface — SELECT bindings, CONSTRUCT output, serializations — speaks native RDF 1.2 reification form: reifiers are real IRIs, rdf:reifies triples are real, and a projected triple-term variable binds a genuine triple term (§7.6). v1's loudly-documented divergence — "an rml:starMap never produces a native triple term, only its basic-encoding stand-in" — no longer holds. A client or portability tool reasoning about query results can treat this engine as an ordinary RDF 1.2 store.

What remains is narrower, and lives entirely below the SQL line, where a relational row genuinely has no atomic triple-term value to hold. The proposition-form encoding (§5) is R2RML-star's own internal wire format for that layer only — modeled on, but not identical to, the W3C RDF 1.2 Interoperability basic encoding (itself non-normative, note-track, Group Note DRAFT). Two deliberate, documented divergences from that note:

  • Deterministic IRIs, not fresh blank nodes. The interop note mints a fresh blank node per encoded proposition; R2RML-star mints a structurally deterministic IRI (§5.1), because a virtualizing engine re-derives the same encoding on every query and needs the same row to always yield the same identifier — a blank node's scope-local freshness would break that invariant across queries.
  • Hybrid visibility. The interop note's basic-encoding triples remain, in principle, ordinarily matchable RDF (it explicitly contemplates hybrid graphs as future work); R2RML-star relies on exactly that — a query can match rdf:PropositionForm/rdf:propositionForm… triples directly if it goes looking, alongside the native answers the query rewrite (§7) normally produces.

R2RML-star still reuses the RML-STAR authoring vocabulary (rml:starMap, rml:quotedTriplesMap) without adopting RML-STAR's own target compiled semantics one-for-one — the role-split identifiers, the deterministic-IRI choice, and the rml:reifierMap extension are R2RML-star's own. An author moving a mapping to another RML-STAR processor should expect the same query answers (the soundness/completeness bar ADR-0032 sets) but not byte-identical internal identifiers.

10.Worked Example

A relational table of census claims, mapped so each age claim is quoted and attributed to a record.

Relational source — census_row 
person_id | age
----------+----
     1042 |  34
     2087 |  29
R2RML-star mapping 
@prefix rr:  <http://www.w3.org/ns/r2rml#> .
@prefix rml: <http://semweb.mmlab.be/ns/rml#> .
@prefix ex:  <http://example.com/> .

<#PersonAge>
    a rr:TriplesMap ;
    rr:logicalTable [ rr:tableName "census_row" ] ;
    rr:subjectMap [ rr:template "http://ex.org/person/{person_id}" ] ;
    rr:predicateObjectMap [
        rr:predicate ex:hasAge ;
        rr:objectMap [ rr:column "age" ]
    ] .

<#PersonAgeAssertion>
    a rr:TriplesMap ;
    rr:logicalTable [ rr:tableName "census_row" ] ;
    rr:subjectMap [ rml:starMap [ rml:quotedTriplesMap <#PersonAge> ] ] ;
    rr:predicateObjectMap [
        rr:predicate ex:assertedBy ;
        rr:objectMap [ rr:constant ex:CensusRecord2026 ]
    ] .

This is byte-for-byte the same mapping v1 authors wrote (§4.1). What changed is what it compiles to. For person 1042's row, the proposition id is the fully-rendered form of each quoted component (§5.1): an I-tagged subject with its template's literal prefix spliced in verbatim, then an L-tagged object — plain, since this mapping's object map carries no explicit rr:datatype for the compiler to fold in as a tag, even though the underlying column is SQL-integer: 

urn:sf-star:pf:http_example_com_hasAge|I|http://ex.org/person/1042|L|34|

and the reifier id is keyed on the declaring map, #PersonAgeAssertion (its slug prefix — abbreviated … below — depends on how the mapping document's base IRI resolves that fragment; what matters is that it is deterministic and distinct per declaring map), over the same two raw columns — the reifier id's own construction is unchanged from v1 (§5.1): 

urn:sf-star:r:…_PersonAgeAssertion|1042|34|

The outer triples map's subject is that reifier id — not the proposition id — linked to it by the injected rdf:reifies triple (§5.3).

Query 1 — bare sugar, asking about the reifier 
PREFIX ex: <http://example.com/>
SELECT ?person ?age ?src WHERE {
    << ?person ex:hasAge ?age >> ex:assertedBy ?src .
}
Result 
?person                     | ?age | ?src
----------------------------+------+---------------------------------
http://ex.org/person/1042   | 34   | http://example.com/CensusRecord2026
http://ex.org/person/2087   | 29   | http://example.com/CensusRecord2026

<< ?person ex:hasAge ?age >> desugars to a fresh-variable rdf:reifies pattern (§7.1–§7.2); ex:assertedBy is asserted on that same reifier (§4.1), so the join resolves without the query ever naming an identifier.

Query 2 — projecting the reifier and the triple term itself 
PREFIX ex:  <http://example.com/>
PREFIX rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#>
SELECT ?r ?t WHERE {
    ?r rdf:reifies ?t .
    ?r ex:assertedBy ex:CensusRecord2026 .
}

Here ?r and ?t are projected directly. ?r binds the plain reifier IRI; ?t — a variable the rewrite tracked as composed throughout (§7.6) — binds a genuine native triple term. In SPARQL 1.2 Results JSON, person 1042's row is: 

{
  "r": { "type": "uri", "value": "urn:sf-star:r:…_PersonAgeAssertion|1042|34|" },
  "t": {
    "type": "triple",
    "value": {
      "subject":   { "type": "uri", "value": "http://ex.org/person/1042" },
      "predicate": { "type": "uri", "value": "http://example.com/hasAge" },
      "object":    { "type": "literal", "value": "34", "datatype": "http://www.w3.org/2001/XMLSchema#integer" }
    }
  }
}

No triple term was ever stored — ?t is reconstructed at the query boundary from the same basic-encoding join Query 1 used — but what the client receives is indistinguishable from a native RDF-star store's own answer.
