RDF-star over your SQL database

Make statements about statements — provenance, certainty, time — and query them with SPARQL-star, without ever copying a row out of your database.

This is the hands-on companion to the R2RML-star specification. It walks from the idea, through a working mapping and query, to the edges you should know about — no prior RDF-star experience assumed.

1.Why quote a triple?

An ordinary triple states a fact:

ex:alice ex:hasAge 34 .

But often you need to say something about that fact — who claimed it, how sure you are, when it was true:

<< ex:alice ex:hasAge 34 >>  ex:assertedBy   ex:CensusRecord2026 .
<< ex:alice ex:hasAge 34 >>  ex:confidence   0.98 .

The << … >> is a quoted triple: a triple used as the subject (or object) of another triple. That is RDF-star. R2RML-star lets you produce and query these directly over relational data.

2.The mental model (30 seconds)

Your database has no column that stores "a triple." So R2RML-star represents each quoted triple as a synthetic id plus four ordinary triples — the RDF 1.2 basic encoding:

?id  rdf:type                     rdf:PropositionForm .
?id  rdf:propositionFormSubject    ex:alice .
?id  rdf:propositionFormPredicate  ex:hasAge .
?id  rdf:propositionFormObject     34 .

Those are just plain triples — exactly what R2RML already maps and what a SQL query already reconstructs. So the whole feature is additive: nothing new happens at execution time, and nothing is materialized.

The key idea You write RDF-star. semantic-fabric compiles it — both the mapping and the query — down to plain triples over your live tables, and compiles the query answer back up. The << … >> never touches your database.

3.Tutorial: your first quoted triple

Goal: model a census table so each age claim can be attributed to the record it came from.

1

Start from a plain mapping

A normal triples map turning each census_row into person ex:hasAge age:

@prefix rr:  <http://www.w3.org/ns/r2rml#> .
@prefix ex:  <http://example.com/> .

<#PersonAge>
    a rr:TriplesMap ;
    rr:logicalTable [ rr:tableName "census_row" ] ;
    rr:subjectMap [ rr:template "http://ex.org/person/{person_id}" ] ;
    rr:predicateObjectMap [
        rr:predicate ex:hasAge ;
        rr:objectMap [ rr:column "age" ]
    ] .
2

Quote it with a star map

Add a second triples map whose subject is the quoted triple from step 1, using rml:starMap → rml:quotedTriplesMap:

@prefix rml: <http://semweb.mmlab.be/ns/rml#> .

<#PersonAgeAssertion>
    a rr:TriplesMap ;
    rr:logicalTable [ rr:tableName "census_row" ] ;
    rr:subjectMap [
        rml:starMap [ rml:quotedTriplesMap <#PersonAge> ]
    ] ;
    rr:predicateObjectMap [
        rr:predicate ex:assertedBy ;
        rr:objectMap [ rr:constant ex:CensusRecord2026 ]
    ] .

That's the whole change. The engine now emits, per row, the four basic-encoding triples plus the ex:assertedBy triple hanging off the same synthetic id.

3

Ask a SPARQL-star question

PREFIX ex: <http://example.com/>
SELECT ?person ?age ?src WHERE {
    << ?person ex:hasAge ?age >> ex:assertedBy ?src .
}
Result 
?person                    | ?age | ?src
---------------------------+------+-------------------------------
http://ex.org/person/1042  | 34   | .../CensusRecord2026
http://ex.org/person/2087  | 29   | .../CensusRecord2026

Under the hood the quoted pattern became a join over the basic-encoding rows. You wrote RDF-star; the database saw ordinary SQL.

4.Asserted vs. non-asserted

Do you also want the plain fact person ex:hasAge age to exist on its own, matchable by ?p ex:hasAge ?a? That's the one knob:

asserted

rml:quotedTriplesMap — the quoted triple and its plain form both exist. Use when the fact is true and you want to annotate it.

non-asserted

rml:nonAssertedTriplesMap — only the quoted form exists; the plain triple is suppressed. Use to talk about a claim without asserting it (e.g. "someone alleged X").

Swap the property; nothing else changes.

5.Quoting in object position

A quoted triple can also be the object — "record R states that ⟨…⟩":

ex:CensusRecord2026  ex:states  << ex:alice ex:hasAge 34 >> .

Put the rml:starMap inside an rr:objectMap instead of the subject map. The engine attaches the four basic-encoding triples to a separate synthetic map keyed on the same id — you don't have to think about that; it just works.

6.Querying: the syntax that bites

SPARQL-star has two spellings of a quoted triple, and it matters to know they're equivalent:

You writeWhat the parser does
<< ?s ?p ?o >> (bare)Rewrites to _:b rdf:reifies <<( ?s ?p ?o )>> internally.
<<( ?s ?p ?o )>> (parenthesized)A quoted triple term, in place.

Both are supported and return the same rows. semantic-fabric recognizes the rdf:reifies form the parser produces and rewrites it to the basic-encoding join.

Why this note exists If a system ignored the rdf:reifies wrapper, every bare-syntax query would silently return nothing — the most dangerous kind of bug, because the query looks fine. semantic-fabric handles it (and is tested for exactly this). Good to know it's the tricky part.

7.What v1 does not do (yet)

These are explicit, honest errors — you get a clear message, never a wrong answer:

  • Nesting — a quoted triple inside a quoted triple.
  • Predicate position — a quoted triple used as a predicate (also disallowed by RDF-star itself).
  • Ground quoted triples in VALUES, the TRIPLE()/isTRIPLE()/SUBJECT()/… functions, quoted terms in a CONSTRUCT template, and a quoted pattern directly on a property-path endpoint.
  • Cross-source quoting — the quoted triples map must share the outer map's table in v1.

Everything else — subject/object position, asserted/non-asserted, variables or constants in any of s/p/o, patterns inside FILTER EXISTS — works.

8.FAQ

Is this real RDF-star, or a workaround?

It's real RDF-star at the surface you write and query. Underneath, semantic-fabric uses the W3C RDF 1.2 basic encoding (plain triples) instead of native triple terms, because a virtualizing SQL engine has nothing to store a triple term in. The specification §9 documents this divergence precisely — it matters if you move mappings to a native RDF-star store.

Does anything get materialized?

No. Both the mapping and the query compile to plain triples / SQL over your live tables. No copy of the data is made.

What does the quoted-triple identity look like if I select it?

A synthetic urn:sf-star:… IRI, deterministic per row (same row → same id, always). It's a stand-in for the proposition, not a native triple term — so isTRIPLE() on it is a v1 boundary rather than returning a possibly-misleading value.

Has anyone done RDF-star over live SQL before?

Per the project's research (ADR-0028 §G, a survey of seven OBDA engines plus the literature): no. This is novel ground — which is why the design is documented so carefully and every boundary is an explicit, tested error.
