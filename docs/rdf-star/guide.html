RDF-star over your SQL database

Make statements about statements — provenance, certainty, time — and query them with SPARQL-star, without ever copying a row out of your database.

This is the hands-on companion to the R2RML-star specification. It walks from the idea, through a working mapping and query, to the edges you should know about — no prior RDF-star experience assumed.

1.Why quote a triple?

An ordinary triple states a fact:

ex:alice ex:hasAge 34 .

But often you need to say something about that fact — who claimed it, how sure you are, when it was true:

<< ex:alice ex:hasAge 34 >>  ex:assertedBy   ex:CensusRecord2026 .
<< ex:alice ex:hasAge 34 >>  ex:confidence   0.98 .

The << … >> is a quoted triple: a triple used as the subject (or object) of another triple. That is RDF-star. R2RML-star lets you produce and query these directly over relational data.

2.The mental model (30 seconds)

Your database has no column that stores "a triple." So under the hood, R2RML-star represents each quoted triple as a synthetic id plus plain triples — the RDF 1.2 basic encoding:

?pf  rdf:type                     rdf:PropositionForm .
?pf  rdf:propositionFormSubject    ex:alice .
?pf  rdf:propositionFormPredicate  ex:hasAge .
?pf  rdf:propositionFormObject     34 .

Those are just plain triples — exactly what R2RML already maps and what a SQL query already reconstructs. So the whole feature is additive: nothing new happens at execution time, and nothing is materialized.

Here's the part that's easy to miss: that encoding is invisible to you. semantic-fabric encodes once, under SQL, and decodes once, right back at the query boundary — so what you get back is a real RDF 1.2 triple term, not the synthetic id. You write RDF-star, and you see RDF-star.

The key idea You write RDF-star. semantic-fabric compiles it — both the mapping and the query — down to plain triples over your live tables, and decodes the answer back up to native RDF 1.2 terms. The << … >> never touches your database, but it's exactly what comes back out.

3.Tutorial: your first quoted triple

Goal: model a census table so each age claim can be attributed to the record it came from.

1

Start from a plain mapping

A normal triples map turning each census_row into person ex:hasAge age:

@prefix rr:  <http://www.w3.org/ns/r2rml#> .
@prefix ex:  <http://example.com/> .

<#PersonAge>
    a rr:TriplesMap ;
    rr:logicalTable [ rr:tableName "census_row" ] ;
    rr:subjectMap [ rr:template "http://ex.org/person/{person_id}" ] ;
    rr:predicateObjectMap [
        rr:predicate ex:hasAge ;
        rr:objectMap [ rr:column "age" ]
    ] .
2

Quote it with a star map

Add a second triples map whose subject is the quoted triple from step 1, using rml:starMap → rml:quotedTriplesMap:

@prefix rml: <http://semweb.mmlab.be/ns/rml#> .

<#PersonAgeAssertion>
    a rr:TriplesMap ;
    rr:logicalTable [ rr:tableName "census_row" ] ;
    rr:subjectMap [
        rml:starMap [ rml:quotedTriplesMap <#PersonAge> ]
    ] ;
    rr:predicateObjectMap [
        rr:predicate ex:assertedBy ;
        rr:objectMap [ rr:constant ex:CensusRecord2026 ]
    ] .

That's the whole change. The engine now emits, per row, four basic-encoding triples describing the quoted (person, age) statement, plus an rdf:reifies triple linking that description to a second id — the reifier — which is what ex:assertedBy actually hangs off. You don't have to manage this split yourself; it's what makes the query in step 3 answerable.

3

Ask a SPARQL-star question

PREFIX ex: <http://example.com/>
SELECT ?person ?age ?src WHERE {
    << ?person ex:hasAge ?age >> ex:assertedBy ?src .
}
Result 
?person                    | ?age | ?src
---------------------------+------+-------------------------------
http://ex.org/person/1042  | 34   | .../CensusRecord2026
http://ex.org/person/2087  | 29   | .../CensusRecord2026

Under the hood the quoted pattern became a join over the basic-encoding rows. You wrote RDF-star; the database saw ordinary SQL.

4.Asserted vs. non-asserted

Do you also want the plain fact person ex:hasAge age to exist on its own, matchable by ?p ex:hasAge ?a? That's the one knob:

asserted

rml:quotedTriplesMap — the quoted triple and its plain form both exist. Use when the fact is true and you want to annotate it.

non-asserted

rml:nonAssertedTriplesMap — only the quoted form exists; the plain triple is suppressed. Use to talk about a claim without asserting it (e.g. "someone alleged X").

Swap the property; nothing else changes.

5.Quoting in object position

A quoted triple can also be the object — "record R states that ⟨…⟩":

ex:CensusRecord2026  ex:states  << ex:alice ex:hasAge 34 >> .

Put the rml:starMap inside an rr:objectMap instead of the subject map. The engine attaches the four basic-encoding triples to a separate synthetic map keyed on the same id — you don't have to think about that; it just works.

Going deeper: nesting

An object-position quote can itself quote another triple, to any depth — "the reifier of ⟨Alice's age⟩ says it comes from ⟨a leaf record's score⟩":

SELECT ?r ?p ?leaf ?score WHERE {
  ?r rdf:reifies <<( ?p ex:hasAge <<( ?leaf ex:hasScore ?score )>> )>>
}

Each level mints its own identifiers and joins bottom-up automatically. The quoted triples map for the inner quote doesn't even need to share a table with the outer one — cross-source quoting works too, by adding rr:joinCondition to the star map.

6.Querying: the matching law

SPARQL-star has two spellings of a quoted triple, and — unlike in v1 — they now ask different questions:

You writeWhat it asks
<< ?s ?p ?o >> (bare)"Who/what reifies this statement?" — desugars to _:b rdf:reifies <<( ?s ?p ?o )>>. Matches only a statement someone actually annotated (a subject-position star map).
<< ?s ?p ?o ~ ?r >> (explicit reifier)Same question, with ?r naming the reifier yourself instead of an anonymous blank node.
?s ?p ?o {| ?pred ?val |} (annotation)Both: asserts ?s ?p ?o and asks who reifies it, in one triple.
<<( ?s ?p ?o )>> (parenthesized)"Is this exact statement, as a value, sitting somewhere?" — matches a triple-term object directly, reified or not.

The two kinds are not interchangeable, and it matters which one you reach for. Take the census mapping from §3: it only ever quotes person ex:hasAge age as a subject (a reifier annotation), never stores it bare as someone else's object. So:

Bare, in object position

SELECT ?p ?age WHERE {
  ?x ex:mentions <<( ?p ex:hasAge ?age )>>
}

Matches nothing for this mapping — nobody's ex:mentions quotes this shape as a plain value.

Reified, as taught in §3

SELECT ?p ?age ?src WHERE {
  << ?p ex:hasAge ?age >> ex:assertedBy ?src
}

Matches — #PersonAgeAssertion is a reifier, and ex:assertedBy hangs off it.

The v1 behavior this replaces v1 treated both spellings as the same thing and let bare syntax match a quoted-triple object too — a real unsoundness (SPARQL 1.2 draws the ReifiedTriple/TripleTerm distinction precisely so this doesn't happen). If you have queries written against v1, audit any bare << … >> pattern matched against an object-position quote — it now returns empty; switch it to the parenthesized form.

7.Querying inside a named graph: GRAPH ?g

If your mapping puts triples into named graphs — a constant rr:graphMap, one per source table, say — you can ask GRAPH ?g { … } and get back which graph each answer came from, not just the answer itself:

PREFIX ex: <http://example.com/>
SELECT ?g ?person ?age WHERE {
  GRAPH ?g { ?person ex:hasAge ?age }
}

?g binds, per row, to the declared graph that triple actually lives in — no quad store, no extra materialization; the engine enumerates the mapping's declared graphs at translate time and joins ordinarily. Two patterns sharing one ?g inside the same block only match when they land in the same graph, for free — it's just an ordinary shared-variable join.

This composes with quoting, too — a star annotation can live inside a named graph, and querying it is completely unremarkable:

PREFIX ex: <http://example.com/>
SELECT ?g ?person ?age ?src WHERE {
  GRAPH ?g {
    << ?person ex:hasAge ?age >> ex:assertedBy ?src .
  }
}

The quoted pattern's basic-encoding join (§3) happens inside the graph pin exactly like an ordinary triple pattern would — the description triples that make the quote answerable are compiled into the same named graph as the quote itself, so this just works, with no extra authoring step.

One boundary to know Property paths (ex:p+, ex:p/ex:q, …) inside GRAPH ?g work when every graph map in your entire mapping is a constant IRI. The moment any triples map anywhere uses a template or column graph map, a path under GRAPH ?g returns a 501 instead of an answer — the graph a row lands in isn't knowable until the query runs, and path translation needs it upfront. Ordinary (non-path) patterns under GRAPH ?g are unaffected, and a fixed GRAPH <g> (not a variable) is unaffected too.

8.Duplicates don't leak through

Your source table almost certainly isn't perfectly deduplicated, and a mapping can end up quoting the same fact twice. Neither should show up as extra rows in a SPARQL-star answer: R2RML's own output is a graph — a set — so a duplicate source row, or two star maps that happen to quote the identical triple, counts once.

semantic-fabric enforces this at query time, at essentially zero cost for the common case. A mapping whose subject is templated on a primary key — the normal shape — needs no extra work at all: the engine proves duplicates are impossible from the key alone and emits the same SQL it always did. Only a mapping that can actually produce duplicates pays for the dedup that guarantees a correct answer.

One thing to watch A non-injective IRI template — one where two different rows can render the exact same IRI — still returns a 501 rather than silently deduping. Add a separator between your template's column references and it becomes injective (and dedup-safe) again. Plain-literal and blank-node templates don't have this restriction; they dedup automatically.

CONSTRUCT follows the same rule: instantiating the same triple from two different rows emits it once, not twice — and so does instantiating it from two different candidate maps that happen to produce the identical triple, not only two rows of the same map.

9.What works now, and what still doesn't

As of ADR-0032, almost everything v1 declared a boundary is implemented:

  • Nesting (object side, any depth — §5), cross-source quoting (§5), and author-controlled reifiers via rml:reifierMap.
  • Ground quoted triples in VALUES, all five TRIPLE()/SUBJECT()/PREDICATE()/OBJECT()/isTRIPLE() functions, =/sameTerm over triple terms, and quoted terms in a CONSTRUCT template (a real triple term comes out; an illegal position is silently dropped, per the SPARQL spec — not an error).
  • Native results — project a reified pattern's identity and you get the reifier's IRI; project a triple-term variable and you get a real triple term back, not a stand-in id.
  • Paths compose now too (2026-07-19): a quoted pattern at a property-path endpoint — and path joins generally, star or not — answer on both query engines, including OPTIONAL and FILTER EXISTS compositions. A path whose predicate is only mapped in a different named graph than the one you're querying now just returns no rows, instead of an error.
  • Mixed unions resolve at the top level, and now under a FILTER too (2026-07-19): a SELECT whose union arms disagree on whether ?t is a triple term — or a VALUES column mixing both — returns each row realized per its own arm. A FILTER touching ?t, sitting at the SELECT top level or inside EXISTS/NOT EXISTS, now resolves the same way instead of erroring.
  • Whole-triple-term equality (?t1 = ?t2) resolves for constant and same-shape template components, and now for differently-shaped IRI and plain-string templates too (2026-07-19); a same-shape pair of different kinds (an IRI template compared against a Literal template, say) now correctly resolves empty too, instead of a wrong match (2026-07-20).
  • Named graphs, including the variable form (2026-07-20): GRAPH ?g { … } now works — see §7 — and composes directly with star annotations declared inside a named graph. Property paths under GRAPH ?g still refuse (501) once the mapping has any non-constant graph map anywhere, even if the path's own predicate never uses one.

What's left is honest, and short:

  • Predicate position — a quoted triple used as a predicate. Not an engine limit: RDF 1.2 itself requires a predicate to be an IRI.
  • Subject-side nesting — a quoted triple whose own subject is another quoted triple. Also spec-impossible, not a scope choice; as a query pattern it matches nothing rather than erroring (see the matching law, §6).
  • Observing a mixed variable outside a qualifying FILTER — a function, equality, or CONSTRUCT template over a variable that's a triple term on one union arm and a plain value on another still errors explicitly, as does a mixed union or VALUES partition nested under a further join. (A FILTER directly at the SELECT top level or inside EXISTS/NOT EXISTS is the one case that now resolves instead of erroring — see §6.)
  • Whole-triple-term equality across typed-literal, language-tagged, or blank-node mismatched shapes, or on any SQL dialect other than PostgreSQL/SQLite/MySQL — a sharpened, explicit error naming the reason. Mismatched IRI and plain-string shapes now resolve on those three dialects; comparing components directly (OBJECT(?t1) = OBJECT(?t2)) always works.

Everything else — subject/object position, asserted/non-asserted, variables or constants in any of s/p/o, patterns inside FILTER EXISTS — works, same as before.

10.FAQ

Is this real RDF-star, or a workaround?

Yes, and now at every surface you touch: what you write, what you query, and what comes back. Underneath, semantic-fabric still uses the W3C RDF 1.2 basic encoding (plain triples) as its own internal wire format, because a virtualizing SQL engine has nothing to store a triple term in at the row level — but it decodes back to native RDF 1.2 at the query boundary, so a result set is indistinguishable from a native RDF-star store's own answer. The specification §9 documents exactly what (narrower) divergence remains, entirely below the SQL line.

Does anything get materialized?

No. Both the mapping and the query compile to plain triples / SQL over your live tables. No copy of the data is made.

What does the identity look like if I select it?

Depends what you project. A reified pattern's identity (?r in ?r rdf:reifies <<(…)>>) is a plain urn:sf-star:r:… IRI, deterministic per declaring star map and row. A triple-term variable (?t in ?r rdf:reifies ?t, or anywhere a <<( … )>> pattern binds one) is a real native triple term — not a stand-in — so isTRIPLE(?t) now genuinely returns true.

What's the difference between << s p o >> and <<( s p o )>>?

The first asks who reifies the statement; the second asks whether the statement itself, as a value, appears somewhere. See the matching law in §6 — they're deliberately not the same question, and (unlike in v1) they can return different rows.

Has anyone done RDF-star over live SQL before?

Per the project's research (ADR-0028 §G, a survey of seven OBDA engines plus the literature): no. This is novel ground — which is why the design is documented so carefully and every remaining boundary is an explicit, tested error or a spec-mandated empty match.
